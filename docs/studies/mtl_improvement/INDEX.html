TL;DR & scope

Goal. The cross-attn MTL regime caps next_region learning regardless of substrate (the regime finding) — MTL trails matched-head STL by −7 to −12pp @ disjoint across 5 states. Fix the regime (the shared-backbone reg pathway + the prior pathway) so that (a) the gap closes toward the STL/composite ceiling and (b) the v14 substrate's STL dual-axis gains finally survive joint training — demonstrated by a regime×substrate interaction (v14 > canonical only after the fix).

Base substrate: check2hgi_design_k_resln_mae_l0_1 (v14). Paper anchor: canonical v11 (RESULTS_TABLE.md §0.1). Recipe: v12 default (B9/H3-alt + log_T-KD ON).

Scale-conditional, not universal

The reg gap and the selector bug are both scale-conditional (selector bug ~1pp at AL/AZ, +7-11pp at FL/CA/TX; v14's Delaunay reg lever is +reg at FL, smaller at small states). The ship recipe in T6 may be state-conditional (a large-state recipe + a small-state recipe). A single universal winner is a stretch outcome, NOT a win condition.

The regime finding — the binding constraint

Every tier exists to attack this. Three independent studies converge:

  1. It is architectural, not interference. mtl-protocol-fix P4 frozen-cat test: with cat fully frozen and cat_weight=0 from epoch 0, MTL reg STILL peaks at ep 2 then crashes. The MTL reg head reads from the shared backbone output; STL reg head reads region embeddings directly through its own STAN backbone → different pathways, different ceilings. (Source: docs/results/mtl_protocol_fix/phase1_phase2_verdict_v6_final.md §F2/Phase 3.)
  2. The structural residual is ~75% of the gap. B9_STL_STAN_SWAP §6.4 (AZ, total gap 10.72pp): softest sharing (CrossStitch) recovers +1.84pp (17%, within σ); recipe ≤0.7pp; thin residual-skip −0.6pp (FALSIFIED); encoder MLP depth flat (EXPERIMENT_NO_ENCODERS); ~75% unexplained. The real STL edge is "the full 9-step raw sequence processed by its own STAN backbone," not a single-step shortcut.
  3. Substrate gains wash out. embedding_eval regime finding: v14 (STL cat 67.36 / reg 0.7024) ≈ canonical in MTL (2-fold seed42 pilot: cat −0.21pp, reg-Acc +0.03pp); routing HGI's STL-winning region tower in also washes out. v14/v13/HGI are all STL-only.

⭐ Per-task disjoint vs joint — MANDATORY for every MTL run (user-requested)

Every MTL result reports both tasks at two epochs, not one:

  • Per-task DISJOINT — each head at its OWN best epoch: cat@best-cat-epoch AND reg@best-reg-epoch. This is the capacity each head reaches; the two epochs differ (reg peaks ~ep 2-4, cat ~ep 30-40 — that divergence is exactly the regime story). Reporting-only (the deployable model is a single checkpoint).
  • JOINT — both heads at the SINGLE deployable checkpoint (the geom_simple selector epoch): cat@joint AND reg@joint. This is the honest one-model-two-predictions deployment number.

So every MTL row is a 2×2: {cat, reg} × {disjoint, joint}. The cat disjoint−joint and reg disjoint−joint gaps quantify how much each task pays for sharing a single checkpoint (the "capacity gap" + "selector bug" from mtl-protocol-fix F1). Plus the STL ceiling per task and the composite ceiling for reg. Example MTL result block:

⭐ Frozen-fold paired design — for architecture/infrastructure experiments (user-requested)

For comparisons where the only intended variable is the model infrastructure (all of Tier 2 architecture, the encoder probe T1.3, head swaps T5), freeze the data partition so fold composition cannot leak into the result:

  • Fix the fold split: all arms in the comparison use the identical StratifiedGroupKFold(groups=userid, seed=S) partition — same users in the same fold across every arm. Run compute_region_transition.py --per-fold --seed S once per seed and reuse the resulting region_transition_log_seed{S}_fold{N}.pt for all arms (the seeded log_T must match the frozen split).
  • Paired Δ per fold: compare arms fold-by-fold (paired Wilcoxon over the shared folds). Because both arms see the identical val users + identical prior, fold-composition variance cancels in the Δ — the residual is the architecture.
  • What it controls: fold-composition / data-difficulty variance (the dominant noise source at small states, σ ≈ 3-4pp on reg). What it does NOT control: init-seed + batch-order variance — for multi-seed runs, vary the seed but keep the fold partition aligned per seed (seed S → its own frozen split + matching log_T). Never pool across mismatched splits.
  • Caveat (F51 fold-1-easiness): frozen folds make paired Δs clean but absolute numbers still carry that partition's easiness — always report Δ vs the frozen-fold baseline, not bare absolutes, for infra comparisons.
  • When NOT to freeze: the multi-seed ship run (T6) and the regime entry gate (T0.3) deliberately vary folds across seeds for generalization — freezing is for the infra-isolation tiers, not the final paper-grade estimate.

Why this order (re-execution-risk minimal)

  1. T0 builds v14 + the measurement frame once. v14 exists only at FL today; the Delaunay reg lever is state-dependent, so building + STL-validating per state is a real prerequisite with a fallback. The three-frontier + fresh-vs-frozen frame, set up once, makes every later tier interpretable.
  2. T1 pins the ceilings (c)+(d) the whole track aims at — before any architecture work, so T2 deltas are anchored. T1.4 then fully HP-TUNES the current heads (α + imbalance losses + the ranked HP axes on the never-leak-free-tuned STL heads) and the tuned result SETS + FREEZES (c)/(d), because T2's gate is measured against them — an under-tuned ceiling would sandbag T2's go/no-go, not just mis-scale a number. The new-architecture search (Tier S) is split off precisely so a late STL win cannot move the baseline T2 was already scored against (the moving-baseline trap).
  3. Tier S runs parallel, not in the chain. It is STL-only (no MTL backbone dependency), so it neither gates nor is gated by T2-T4 — it branches off T1.4 and rejoins at T5 as a candidate-head supplier. It is placed in document order between T1 and T2 (where the user expects it) but executes concurrently with Tier 2-4 on the box. Framing is ceiling/baseline integrity, NOT MTL transfer (the regime finding predicts STL gains wash out) — so Tier S is positive-EV via a better/firmer ceiling and a reviewer-proof baseline, with any MTL transfer a weak bonus that T2 already chases.
  4. T2 is the structural change (P4: the gap is the shared-backbone pathway). It conditions everything downstream — the prior pathway, loss, LR, and heads all behave differently under a dual-tower than under cross-attn. The load-bearing T2.1 runs on BOTH substrates to prove the regime×substrate interaction (advisor-mandated — you cannot show "fixing the regime makes the substrate matter" on v14 alone).
  5. T3 prior pathway after arch — log_T-KD is confirmed (+2.4-5pp) and already default; the open question is whether it stacks with the dual-tower or is redundant.
  6. T4 gated, T5 terminal — loss is low-EV (P4), so it only runs against a residual; head re-ablation is last because the best head depends on the consolidated backbone+prior.

Execution environment & parallelization (i9-14900K + A40 45GB)

The runs execute on a single box: Intel i9-14900K (24 cores / 32 threads) + NVIDIA A40 (45 GB usable). Observed timing (5f × 50ep, bs=2048, from the 2026-06-03 manifest): FL MTL ≈ 14 min/seed; AZ ≈ 3 min; AL ≈ 1.6 min. The MTL runs are cheap; the substrate builds (design_k, 500 epochs on GPU) are the heavy serial spine. The current drivers (scripts/_v14_run/) run everything serially — that is the parallelization headroom.

The parallelization plan (smart, throughput-safe)

  1. GPU — collocate small jobs via CUDA MPS, bounded by an aggregate-VRAM budget. Small-state MTL runs (AL/AZ/GE, ~2-3 min, light) underutilize the A40 — time-sliced they sit at ~20% util; under MPS their kernels overlap across SMs (~60% util, up to ~4× collocated throughput — refs). Run the 4 seeds of a small/middle state concurrently under MPS (one-run wall instead of 4×). FL is heavier (closer to SM-saturating) → collocate only 2. Concurrency cap = min( ⌊0.85 × 45GB / per-run-peak-VRAM⌋ , SM-sensible cap ) — measure per-run peak VRAM once (one MTL run), then set the cap; reserve ~15% headroom (OOM + fragmentation). Builds saturate the GPU → one build at a time, never collocated.
  2. CPU — parallelize prep + analysis on the 32 threads, overlapped with GPU work. Seeded per-fold log_T (compute_region_transition.py), postbuild, aggregation, and the analysis/probe scripts are CPU-bound — run them with xargs -P/GNU-parallel across seeds×folds×states while the GPU is busy with builds/MTL. Pin per-process threads (OMP_NUM_THREADS = 32 / N_concurrent) so collocated GPU processes don't thrash the CPU.
  3. Pipeline the heavy spine. While state N's substrate builds on the GPU, run the (CPU) log_T builds + the (cheap, MPS) small-state MTL seeds for already-built states. The build queue is the critical path; everything else fills its gaps.
  4. Tier S is the prime collocation filler. The deep STL search (Tier S) is STL-only and off the linear chain → run its seeds/states concurrently under MPS with the Tier 2-4 MTL runs. STL reg/cat single-task jobs are lighter than the joint MTL runs, so they slot into the GPU's spare SMs at near-zero added wall-clock. This is exactly why "full in-track deep tier" is affordable: Tier S overlaps the architecture tiers rather than extending the schedule.
  5. Don't-hurt rules. (a) cap by the VRAM budget with headroom — never fill 45GB; (b) one GPU build at a time (builds are compute-saturating — collocating them gives ~zero speedup + OOM risk); (c) FL collocation ≤2 (compute-bound — more concurrency ≈ no wall reduction); (d) each concurrent process writes its OWN run dir + reads its OWN seeded log_T — the driver's stage_logT copy step must complete before concurrent reads (race risk); (e) seeds stay deterministic under collocation (each process its own seed); (f) if MPS is unavailable, fall back to a small fixed process pool (2-3) — still better than serial for small states.
