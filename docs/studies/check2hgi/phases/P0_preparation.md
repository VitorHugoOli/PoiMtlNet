# Phase P0 — Preparation

**Goal:** have integrity-checked data, simple-baseline reference numbers, and resolved audits before any scientific test starts. P0 is the "known-good reference" phase — everything downstream compares against the simple-baseline floor computed here.

**Duration:** ~3h (integrity + code-deltas + baselines + audits).

**Embedded claims tested:**
- **CH04** — Simple baselines (floor that every learned model must beat).
- **CH05** — Ranking metrics vs macro-F1 (baseline-vs-learned comparisons document this).
- **CH14** — Preprocessing shortcut audit (both code inspection AND unconditional fclass-shuffle ablation).
- **CH15** — Transductive embedding-leakage audit (one held-out-user-fold retrain, upper-bound estimate).

### P0.2 — Label round-trip spot check

Verify that the labels generated by the `next_poi` and `next_region` pipelines correspond to the actual next-POI / next-region in the user's check-in stream. Catches off-by-one, placeid_to_idx inversion, and index-vs-label confusion bugs that PI.3 (label range only) misses.

**Procedure:** Hand-pick 20 sequences from each state's `next_poi.parquet`. For each sequence:
1. Read `poi_idx` column.
2. Use the *inverse* `idx_to_placeid` dict (derived from `placeid_to_idx` in `checkin_graph.pt`) to get the raw placeid.
3. Look up the user's check-in stream in `data/checkins/<state>.parquet`, find the user, confirm that the placeid immediately following the 9-window IS that placeid.

All 20 must round-trip. A single mismatch pauses the coordinator.

Same check for `next_region.parquet` (poi_idx → poi_to_region → region).

### P0.5 — Simple baselines (floor / "known-good reference")

**Purpose:** establish the floor every learned model must beat. Catches pipeline bugs (if learned Acc@10 is not 2× above these, something is broken). **This is the closest equivalent to fusion's P0.4 CBIC replication.**

**Baselines computed per state, per task (next_poi AND next_region):**

| Baseline | Definition | Compute cost |
|---|---|---|
| **Random** | Uniform draw over all labels. Acc@K = K / n_classes. | Closed-form; no data pass. |
| **Majority-class** | Always predict the most frequent label in train fold. Acc@1 = majority fraction; Acc@K = cumulative top-K. | One pass. |
| **1-step Markov** | P(next | current): for each sequence, predict argmax over `P(next=i | current=last_poi_in_window)` learned from train. | One pass train + one pass val. |
| **Top-K popular** | Always return top-K most frequent labels. Acc@K = cumulative fraction. | One pass. |
| **User-history top-K** | Return top-K labels the user has visited in train. Handles "return to known POI" pattern. | Per-user lookup. |

**Specification for the user's implementation branch:**
- New script `scripts/compute_simple_baselines.py` that takes `--state <s> --engine check2hgi --task {next_poi,next_region}` and writes `docs/studies/check2hgi/results/P0/simple_baselines/<state>/<task>.json`.
- For each baseline, report `acc1, acc5, acc10, mrr` on the val set of each of the 5 frozen folds.
- Aggregate: mean + std across folds.
- Summary JSON includes the computed majority-class fraction (used by `val_joint_geom_lift` in the runner).

**Expected ballpark numbers** (for integrity check that the script works):
- Next-POI, AL, random: Acc@10 ≈ 10 / 11,848 ≈ 0.08%.
- Next-POI, FL, random: Acc@10 ≈ 10 / 76,544 ≈ 0.013%.
- Next-region, AL, majority: Acc@1 ≈ 2.3% (matches the PI.4 distribution warning).
- Next-region, FL, majority: Acc@1 ≈ 22.5%.
- 1-step Markov and user-history top-K should substantially beat majority (users have strong sequential patterns). If they don't, flag.

### P0.6 — Shortcut audit (CH14, **unconditional**)

**Step 1 — code inspection.** Read `research/embeddings/check2hgi/preprocess.py` and `check2hgi.py`. Look for:
- POI2Vec usage (the specific fclass-identity mechanism fusion's C29 identified in HGI).
- Any direct use of `fclass` or `category` columns as hard features.
- Any POI-identity preservation channel (direct placeid embedding lookup on the MI loss path).

Document findings in `results/P0/ch14_audit.md`.

**Step 2 — unconditional fclass-shuffle ablation** (per review agent's §2.3 recommendation):

Run single-task next-POI on Alabama with `fclass` column shuffled in preprocessing (same mechanism as fusion's `scripts/hgi_leakage_ablation.py` arm C). 1 fold, 10 epochs (quick sanity).

**Compare:** next-POI Acc@10 drop between baseline and shuffled preprocessing.

- If drop < 3 pp → **CH14 confirmed** (no fclass shortcut in check2HGI next-POI prediction).
- If drop is large (say > 10 pp) → **CH14 refuted by construction**; the representation relies on the fclass shortcut. Paper pivots: report the shortcut as a finding, propose a mitigation.
- If drop is moderate (3–10 pp) → **CH14 partial**; document as a caveat.

### P0.7 — Transductive-leakage audit (CH15)

**Purpose:** bound the magnitude of the upstream leakage where Check2HGI is trained on 100% of check-ins (including val-fold users) before the downstream task holds users out.

**Procedure:**
- Use fold 0 of the frozen StratifiedGroupKFold split.
- Retrain Check2HGI WITHOUT fold-0 val users' check-ins (i.e. exclude their trajectories from the pretraining data).
- Run single-task next-POI on fold 0 with the held-out-trained embeddings vs the standard embeddings.
- Compare Acc@10.

**Decision:**
- `|Δ Acc@10| < 1 pp` → **CH15 bounded**; declare the transductive effect as small, document in paper limitations.
- `|Δ Acc@10| > 2 pp` → **CH15 significant**; the paper has a bigger limitation section, and we need to decide whether to retrain Check2HGI per-fold for the final numbers (very expensive).

Cost: ~20 min for the Check2HGI retrain + ~10 min for the single-task run on one fold. Run once as an upper bound; if bounded, cite on all CH01/CH02 results.
