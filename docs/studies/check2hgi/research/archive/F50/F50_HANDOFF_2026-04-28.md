## 3 · The folds.py bugfix — load-bearing context for new agents

`src/data/folds.py` had a hardcoded gate that publishes the `last_region_idx` aux tensor only when `task_b.head_factory == "next_getnext_hard"`. New GETNext-hard variants (e.g. `next_getnext_hard_hsm`, future hierarchical-softmax variants, future autodiff-detach variants) will silently fail with **0% reg F1** unless added to the gate.

**Symptom of this bug:** training runs technically succeed but the reg progress bar shows `tr=N0.00|C…` (or any tiny non-zero) for *all* epochs while cat trains normally. Validation reg accuracy ≈ 1/n_regions (random uniform).

**Fix:** add the new head's name to `_HEADS_REQUIRING_AUX_MTL` set in `src/data/folds.py:~894`. Keep in sync with `_HEADS_REQUIRING_AUX` in `scripts/p1_region_head_ablation.py:~96`.

**Verification recipe** (always run before a 5-fold MTL run):

```bash
PYTHONPATH=src DATA_ROOT=... OUTPUT_DIR=... \
  python scripts/train.py \
    --task mtl --task-set check2hgi_next_region \
    --state alabama --engine check2hgi \
    --model mtlnet_crossattn --mtl-loss static_weight --category-weight 0.75 \
    --cat-head next_gru --reg-head <NEW_HEAD> \
    --reg-head-param ... \
    --task-a-input-type checkin --task-b-input-type region \
    --folds 1 --epochs 2 --seed 42 --batch-size 1024 \
    --scheduler constant --cat-lr 1e-3 --reg-lr 3e-3 --shared-lr 1e-3 \
    --no-checkpoints --no-folds-cache
```

After 2 epochs on AL, expect reg F1 ≥ 0.5% and reg accuracy ≥ 5%. If both stay 0, the aux gate is the first thing to check.
