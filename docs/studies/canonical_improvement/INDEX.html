TL;DR & scope

Goal: close the cat F1 / reg Acc@10 / fclass-probe Pareto gap that canonical Check2HGI leaves open, using only mechanisms native to c2hgi training — no HGI pretrained artifacts, no fclass labels in the encoder (fclass is the linear-probe target — using it as a feature or supervision is a tautological leak), no merge-family designs. The track is structured so its surviving improvements can be later stacked on the merge family (B/M) in a future synthesis study.

Out of scope (covered by ../merge_design/, do not duplicate):

  • Importing HGI's frozen POI2Vec at the POI‑pool boundary (Designs B/H/I/J/M)
  • POI2Vec in the check-in input (Design E / A)
  • Heterograph with POI nodes (Design D)
  • POI↑POI contrastive boundary on the merge substrate (Lever 6)

What this track does: 18 experiments across 5 tiers attacking the canonical engine's weaknesses — recipe hygiene, p2r-boundary loss shape (asymmetrically untested), encoder inductive bias, semantic-recovery via unsupervised proxies, and one stretch tier (Tier 5) for native POI-identity mechanisms.

Hard rules

  • No merge-family code paths. Do not import output/hgi/<state>/poi_embeddings.csv or any HGI artifact. Adding a feature derived from HGI's training output is forbidden; deriving from intrinsic data (OSM category, coordinates, visit counts) is fine.
  • No fclass-as-feature. fclass is the linear-probe label. Using it in the encoder — even as a learned embedding side-feature — trivially passes the probe and tells you nothing. Reframe to: category (coarse 7-class) supervision, unsupervised prototypes, or fclass-correlated proxies (popularity, opening-hours profile, co-visit category-mix).
  • Always pre-flight T1.1 (leak audit) on canonical first. Establishes the linear-probe floor for every later comparison.
  • Re-run baselines at comparison time. Do not quote pre-Phase-11 numbers as canonical. The FL +13 pp stale-baseline incident in merge_design is the cautionary tale.
  • Statistical claims need n ≥ 10. At n=5 the Wilcoxon floor is p=0.0312. Any "wins HGI on reg" claim requires multi-seed (T1.2) to be in place first.

Evaluation framework

Primary metrics (canonical protocol, must preserve)

  • Next-category macro F1 — next_gru head, B9 MTL recipe (alternating-SGD, Cosine LR, per-head LRs cat 1e-3/reg 3e-3/shared 1e-3, static_weight cat=0.75).
  • Next-region Acc@10 — next_getnext_hard head, per-fold leak-free region_transition_log_seed42_fold{1..5}.pt.

Generality probes (must report all five for every experiment)

  1. POI fclass linear probe — macro-F1 of logistic regression on frozen POI embeddings → fclass label. Canonical floor: 4 %. HGI ceiling: 98 %. Reuse scripts/probe/generality_probes.py.
  2. POI kNN-Jaccard@10 vs HGI — geometric similarity to HGI's POI2Vec. Canonical: 0.006.
  3. Region category-mix regression R² — predict per-region category-fraction vector from region embedding.
  4. POI popularity regression R² (new) — predict log(visit_count) from POI embedding. Detects popularity collapse.
  5. Temporal autocorrelation (new) — same-POI, disjoint-time-bin cosine. High → check-in collapses to POI prototype (canonical's design intent is the opposite).

New eval probes to wire up (currently missing)

  • Next-POI top-k accuracy (Hit@1/5/10, MRR) — the axis flagged missing in merge_design/AUDIT_HGI_GAP.md §3. Use the existing scripts/p1_poi_head_ablation.py (fixed in commit e99e904: `OneCycleLR` now wired in + per-step `scheduler.step()` + `label_smoothing=0.1`). The existing T1 next-POI JSONs in results/P1/poi_head_*_T1_*_5f50ep.json were trained under the pre-fix suboptimal regime (fixed LR, no label smoothing) and are pessimistic absolute numbers — do NOT reuse them as canonical-improvement baselines.   • User-trajectory reconstruction — mask 15 % of a user's check-ins, decode via kNN over learned embeddings, measure recall@k.

Statistical gates

  • TOST non-inferiority on cat F1, δ=2 pp (same gate as merge_design).
  • Wilcoxon one-sided strict p<0.05 on reg vs canonical baseline.
  • Any "beats HGI" claim requires n ≥ 10 (multi-seed via T1.2, otherwise p-floor blocks).
  • Both axes simultaneously: at n=5 no design clears it; at n ≥ 10 the gate is reachable.

Mandatory pre-flight

Tier 1.1 (leak audit) must pass on canonical first. Mirror the Design D diagnostic (scripts/probe/leak_sniff_ijm.py): linear probe on the last check-in's embedding alone. Canonical should land around 30 % (the harmless baseline leak from sequence edges). STL vs MTL training protocol — clarification

Default: STL paired tests for Tier 1–4 winners; B9 MTL only for final shipping candidates. The B9 MTL recipe (alternating-SGD, Cosine LR, per-head LRs, static_weight cat=0.75) approximately doubles compute and is the protocol used for shipping numbers; merge_design's headline paired tests were largely STL. Run Tier 1–4 experiments under STL with the matched-head heads (next_gru for cat F1, next_getnext_hard for reg Acc@10), 5 folds × 50 epochs, single seed 42. Promote the winning Tier-4 candidates to B9 MTL at the end of Tier 4 for the final shipping-grade comparison. This matches merge_design's protocol and keeps statistical comparisons across the two tracks consistent.

AL/AZ → FL promotion protocol

Most experiments specify AL+AZ as the first run. Promote a winning experiment to FL only if (a) cat F1 non-inferior at TOST p<0.01 at both AL+AZ, AND (b) reg Acc@10 strictly positive at Wilcoxon p<0.05 at both. Always re-run a fresh FL canonical baseline under matched protocol at the time of comparison — never quote pre-existing FL canonical numbers. The merge_design FL stale-baseline incident (+13 pp inflation from an old protocol baseline) is the cautionary tale.
