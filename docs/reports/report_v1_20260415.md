## 1. Contexto: por que recomeçamos do zero

Durante a preparação do artigo para o BRACIS 2026, identificamos um erro de rotulação no dataset utilizado em todos os experimentos anteriores — incluindo os resultados apresentados no CBIC 2025. Algumas categorias de POIs estavam associadas a labels incorretos, o que distorceu todas as comparações: tanto os deltas internos (fusão vs. HGI, MTL vs. tarefa única) quanto as comparações com baselines publicados eram feitas em condições incompatíveis.

**O que foi o bugfix:** O pipeline de pré-processamento atribuía a cada POI uma categoria a partir de um mapeamento de categorias do Foursquare/Gowalla que estava parcialmente incorreto — certos tipos de local (ex.: parques, lojas de conveniência) recebiam labels de outra categoria. Como o modelo aprende a separar as classes, treinar e avaliar com labels errados distorce tanto as métricas absolutas quanto qualquer comparação relativa (entre modelos, embeddings ou datasets). A correção envolveu regenerar o mapeamento de categorias e reprocessar todos os inputs a partir dos dados corrigidos.

A decisão foi regenerar os embeddings e todas as entradas do modelo a partir dos dados corrigidos, e reiniciar o estudo de ablação. Os resultados anteriores estão arquivados em `docs/archive/` como referência metodológica, mas **não podem ser usados como evidência científica**.

### 1.1 Segundo bug metodológico: NashMTL rodava em modo equal-weight

Este é um problema **separado e independente** do bug de rotulação — e afeta diretamente a interpretação dos achados sobre otimizadores (tópico 4).

**O que aconteceu:** A biblioteca `cvxpy` usada pelo NashMTL depende de um solver externo chamado ECOS. Esse solver não estava instalado no ambiente onde os experimentos pré-bugfix foram rodados. O código continha um `except` genérico (*bare-except*) que silenciava o erro e fazia o NashMTL cair silenciosamente para pesos iguais `[1, 1]` — comportamento idêntico ao `EqualWeight`. Em outras palavras, **todos os experimentos que acreditávamos estar usando NashMTL estavam, na prática, usando ponderação igual** — o mesmo baseline contra o qual comparávamos os métodos adaptativos.

**Consequência direta sobre os achados:** O tópico 4 conclui que "ponderação igual supera métodos adaptativos". Isso é verdade — mas a comparação inclui um "NashMTL" que era equal-weight disfarçado. A robustez qualitativa do achado (métodos adaptativos não ajudam neste problema) provavelmente sobrevive, pois PCGrad, GradNorm, FAMO etc. foram testados com seus solvers próprios. Mas o específico resultado do NashMTL precisa ser revalidado com o solver funcionando.

O bug foi corrigido com a adição de detecção explícita do solver. Testes automatizados validam que o NashMTL agora efetivamente altera os pesos entre iterações.

> **Em aberto:** O prazo original do BRACIS era 20/04/2026. Com o recomeço forçado, a submissão precisa ser reavaliada — publicação parcial (P0–P2) ou redirecionamento para journal?

## 2. Protocolo de avaliação: vazamento de dados identificado e corrigido

**O problema:** O código anterior usava uma divisão chamada *StratifiedKFold*, na qual o **mesmo usuário podia aparecer tanto no treino quanto na validação**. O modelo "via" os padrões de mobilidade de um usuário durante o treino e depois era avaliado sobre esse mesmo usuário — uma vantagem artificial que inflava as métricas.

**A correção:** O código atual usa *StratifiedGroupKFold*, garantindo **isolamento total de usuários**: cada usuário aparece exclusivamente no treino ou na validação, nunca nos dois. Este é o protocolo correto para medir generalização real para novos usuários.

**Quanto isso importava:** Ao comparar os dois protocolos mantendo tudo o mais igual, a diferença foi de **~1,3 pontos de F1 no eixo next-POI** (35,9% com vazamento vs. 34,6% sem). Cerca de 75% da diferença entre os resultados reportados no código antigo (~36%) e os do código novo (~34%) era artificial.

**Por que isso importa para o artigo:** Qualquer comparação com baselines da literatura exige protocolo consistente. Os números absolutos caem levemente, mas são os únicos defensáveis numa revisão.

## 2.1 Novo achado (15/04): Category F1 mede fidelidade de fclass, não representação aprendida

Durante a auditoria do pipeline HGI identificamos um **atalho taxonômico no dataset** que muda a interpretação de uma das duas métricas principais do estudo. Não é vazamento clássico (nenhum rótulo de validação escapa para o treino), mas o impacto para o artigo é equivalente.

### O que foi descoberto

Cada POI do Gowalla tem duas colunas herdadas do OpenStreetMap (OSM): `fclass` (sub-tipo fino, ~300 valores como "Burgers", "Motel", "Church") e `category` (classe grossa, 7 valores como Food, Travel, Community). O mapeamento **`fclass → category` é 100% determinístico** em todos os 6 estados avaliados: "Burgers" é sempre Food, "Motel" é sempre Travel, nunca se cruzam.

O POI2Vec, pré-etapa do HGI, gera embeddings **no nível do fclass** — ou seja, todos os POIs com o mesmo fclass recebem o mesmo vetor de entrada. Como fclass determina category univocamente, qualquer modelo que aprenda a distinguir fclasses no espaço vetorial **automaticamente** resolve a tarefa de Category — sem nunca ter visto um rótulo de category durante o treino. A informação do target já está estruturalmente codificada na entrada.

### Analogia das frutas

Imagine treinar um modelo para classificar frutas como "doce" ou "azeda" e passar como *feature* o nome da fruta ("maçã", "limão", "morango"). O modelo acerta 99%. Agora embaralhe os nomes — o que era "maçã" vira "limão" aleatoriamente. O modelo cai para 50%. Isso **não prova** que ele aprendeu a identificar doçura; prova que **apenas memorizou** um dicionário nome→sabor. A feature carregava a resposta de graça.

No nosso caso: `fclass` é o "nome da fruta", `category` é "doce/azeda". O modelo de Category nunca aprendeu sobre o espaço semântico dos POIs — apenas consultou o dicionário fclass→category que está implícito no embedding.

### O experimento decisivo (braço C da ablação)

Permutamos a coluna `fclass` entre os POIs (com `category` intacto), rodamos todo o pipeline do zero e treinamos o MTL. Resultado em Alabama (1 fold, seed 42):

| métrica | baseline | shuffle fclass | queda |
|---|---|---|---|
| Category F1 | 0.7855 | **0.1437** | **−64,19 p.p.** |
| Next-POI F1 | 0.2383 | 0.1988 | −3,95 p.p. |

0.1437 é essencialmente o chute aleatório para 7 classes (1/7 ≈ 0.143). O Category virou random; o Next-POI mal se abalou.

### O que precisa mudar no artigo

**Métricas e enquadramento:**

1. **Next-POI F1 passa a ser a métrica-âncora** para qualquer afirmação sobre "qualidade de representação aprendida". É a única tarefa que não admite o atalho do fclass.
2. **Category F1 vira sanity check** de fidelidade do embedding. Comparações cross-engine em Category F1 (HGI vs. DGI vs. Fusão) devem ser renomeadas para *"preservação de fclass num espaço de 64 dimensões"* — ainda informativas, mas medem compressão, não aprendizado espacial.
3. **Joint F1 precisa de escrutínio.** Como ele pondera Category e Next-POI, herda parcialmente o atalho do fclass. Duas opções: (i) reportar as duas métricas separadamente e evitar joint como evidência primária, (ii) recalcular com peso menor em Category, ou (iii) substituir por uma métrica que não incorpore Category como evidência de representação.
4. **Parágrafo de caveat é obrigatório** na seção de avaliação: reconhecer o determinismo fclass→category, citar o braço C, reenquadrar Category F1 como sanity check. Qualquer revisor que conheça OSM fará essa objeção de imediato — tratá-la proativamente fortalece o rigor do trabalho.

**Reposicionamento narrativo:**

5. A narrativa sai de *"MTL bem configurado com HGI aprende representações espaciais/semânticas ricas"* para algo mais sóbrio: *"para a tarefa de Next-POI prediction (a única não-trivial neste dataset), investigamos sistematicamente MTL e fusão de embeddings, controlando para o atalho taxonômico do OSM"*. É reposicionamento, não abandono — e a transparência do caveat fortalece o trabalho em revisão.

### O que isso NÃO significa

- Não é leakage clássico no código. `data.y` nunca é lido durante o treino do HGI/POI2Vec (verificado por auditoria linha-a-linha).
- Não é defeito específico do HGI. Qualquer engine que indexe embeddings a partir do fclass (DGI, Check2HGI, etc.) tem a mesma propriedade.
- Não invalida os números absolutos já coletados — apenas muda sua interpretação.

> **Próximo passo:** replicar o braço C em Florida antes da submissão BRACIS (~15 min) para confirmar que a pureza fclass→category = 1.0 se mantém lá (já verificada em tabela de pureza dos 6 estados).

## 3. Configuração do modelo: arquitetura e hiperparâmetros

**Contexto:** A arquitetura do MTLnet ainda não está finalizada — as fases P1–P4 do estudo incluem uma ablação sistemática de arquiteturas, otimizadores e cabeças. O que foi realizado até aqui são testes de orientação para garantir que o código novo pelo menos está nivelado com o antigo.

**Achado de cabeça:** O modelo MTL possui duas saídas especializadas ("cabeças") — uma para categoria de POI e outra para next-POI. Identificamos que a cabeça de categoria padrão do código novo havia mudado inadvertidamente de um **ensemble de caminhos paralelos** (versão anterior) para uma arquitetura baseada em **Transformer**. O teste direto mostrou que o ensemble é +1,1 ponto melhor em categoria sem custo para next-POI. O ensemble foi restaurado como padrão atual; a escolha final de cabeça será confirmada pela ablação completa.

**Achado de hiperparâmetros:** Testamos quatro combinações de batch size (2048 vs. 4096) e norma de gradiente (1,0 vs. 2,2 — o valor do código antigo). Nenhuma combinação superou o padrão atual para next-POI. A configuração atual está bem calibrada; não há ganho em reverter para os valores anteriores.

| Configuração | F1 next-POI | F1 categoria |
|---|---|---|
| batch=4096, norma=1,0 (padrão atual) | **34,6%** | 76,0% |
| batch=4096, norma=2,2 | 34,2% | 76,6% |
| batch=2048, norma=1,0 | 34,3% | 76,5% |
| batch=2048, norma=2,2 | 34,2% | 76,6% |

> **Em aberto:** A escolha definitiva de arquitetura (CGC, DSelectK, MMoE vs. base) e de cabeças depende das fases P1–P3, que ainda não foram executadas nos dados corrigidos.

## 4. Achado pré-bugfix: otimizadores complexos não superam ponderação simples

*Este achado vem dos experimentos anteriores ao bug. A tendência é robusta o suficiente para ser discutida, mas **precisa ser revalidada nos dados corrigidos** — os números absolutos e os rankings relativos podem mudar. Consideramos este ponto em aberto, assim como o ponto 3.*

**O que foi testado:** 10 métodos adaptativos de balanceamento de tarefas (NashMTL, PCGrad, GradNorm, FAMO, BayesAggMTL, entre outros) foram comparados com a estratégia mais simples possível: pesos iguais para as duas tarefas (sem adaptação alguma).

**Resultado:** A ponderação igual foi a melhor ou ficou estatisticamente empatada com todos os métodos adaptativos testados. Dois métodos modernos (FAMO, 2023; BayesAggMTL, 2024) **pioraram** o desempenho em ~3 pontos.

| Abordagem | Referência | Joint Score |
|---|---|---|
| Pesos iguais | — | **0,301** |
| db_mtl | 2024 | 0,303 |
| fairgrad | 2024 | 0,310 |
| NashMTL | 2022 | 0,300 |
| FAMO | 2023 | 0,270 |
| BayesAggMTL | 2024 | 0,268 |

**Por que isso acontece:** A explicação está na natureza das tarefas. Métodos adaptativos são projetados para resolver *conflito de gradientes* — situações onde as duas tarefas "puxam" o modelo em direções opostas. Neste problema, o conflito entre prever categoria e next-POI é **quase zero**. Sem conflito real para resolver, os mecanismos adaptativos introduzem ruído sem trazer benefício.

**Relevância para o artigo:** Isso estende para métodos de 2023–2024 o resultado de Xin et al. (NeurIPS 2022), que já havia mostrado que métodos adaptativos de até 2022 não ajudavam. É uma contribuição direta e publicável.

## 5. Achado pré-bugfix: hierarquia de importância dos componentes

*Revalidação necessária nos dados corrigidos.*

A ablação sistemática revelou uma hierarquia clara de impacto no desempenho:

> **Embedding > Arquitetura de compartilhamento > Otimizador > Escolha de cabeças**

O componente que mais determina o teto de desempenho é a **qualidade dos embeddings de entrada**. A arquitetura de como as tarefas compartilham informação vem em segundo. O otimizador de balanceamento mal aparece. Essa hierarquia foi consistente em Alabama e Florida.

Um exemplo concreto: nenhuma arquitetura testada com embeddings DGI conseguiu igualar a configuração mais simples com embeddings HGI. A diferença entre as duas famílias de embeddings era de ~45% no score combinado.

**Relevância para pesquisadores de MTL:** boa parte da literatura de MTL investe esforço em otimizadores mais sofisticados. Este resultado sugere que, para predição de POI, o retorno do investimento está muito mais em representação do espaço de entrada do que na estratégia de balanceamento de gradientes.

## 6. Achado pré-bugfix: arquitetura de compartilhamento importa — CGC supera FiLM

*Revalidação necessária nos dados corrigidos. Adicionalmente, a comparação entre arquiteturas faz parte das fases P1–P2 do estudo atual — este resultado **será formalmente revisitado** antes de qualquer conclusão definitiva.*

Dentro da camada de compartilhamento entre tarefas, testamos quatro estratégias:

| Arquitetura | Mecanismo | Joint Score |
|---|---|---|
| **CGC** (melhor variante) | Roteamento por especialistas | **0,430** |
| DSelectK | Seleção aprendida de especialistas | 0,422 |
| MMoE | Mixture of experts | 0,395 |
| MTLnet base | FiLM (modulação multiplicativa) | 0,371 |

A diferença entre a melhor e a pior é de **16%** no score combinado. A arquitetura *Customized Gate Control* (CGC), que usa uma mistura de especialistas compartilhados e específicos por tarefa, foi consistentemente a melhor no dataset com embeddings HGI.

**O mecanismo:** O FiLM (base) apenas *escala e desloca* as features existentes — tem expressividade limitada para condicionar o processamento por tarefa. O CGC *seleciona e combina* saídas de redes especializadas, permitindo representações verdadeiramente específicas por tarefa. A melhoria não se explica apenas pelo número de parâmetros (CGC tem 1,6x mais parâmetros que o base, mas melhora 16% — indicando benefício estrutural).

**Detalhe interessante:** a melhor arquitetura muda com a qualidade do embedding. CGC ganha com embeddings HGI (fortes); DSelectK ganha com DGI (fracos). Hipótese: quando os embeddings são fracos, os mecanismos de seleção mais expressivos do DSelectK compensam o sinal de entrada; quando o sinal já é forte, a complexidade extra do DSelectK vira overhead.

## 7. Observação preliminar: avaliação isolada de cabeças pode não refletir o desempenho no MTL

*Revalidação necessária nos dados corrigidos. Este é um ponto exploratório — **não deve ser tratado como conclusão**.*

**O experimento:** Avaliamos algumas cabeças de rede isoladamente (treinando diretamente sobre os embeddings brutos, sem o backbone MTL). Depois, inserimos essas cabeças no modelo MTL completo.

**A observação:** Em ao menos um conjunto de testes, o ranking de cabeças avaliadas isoladamente não se manteve no modelo MTL completo — cabeças que pareciam melhores no isolamento não necessariamente ganharam no contexto MTL.

**Limitações importantes desta observação:**
- A arquitetura final do MTLnet ainda não estava definida quando esses testes foram feitos; variações nos hiperparâmetros das cabeças produziram melhorias relevantes em outros experimentos, indicando alta sensibilidade que não foi explorada de forma controlada.
- Não havia baselines claras para definir o que seria uma "boa cabeça" no contexto MTL.
- O número de combinações testadas foi pequeno — não é possível excluir que a queda de desempenho se deva a incompatibilidade de hiperparâmetros, e não ao fenômeno de co-adaptação em si.

A hipótese de que backbone e cabeças se co-adaptam durante o treino MTL é plausível, mas **a evidência atual é insuficiente** para sustentá-la como achado formal. Será revisitado nas fases P2–P3 do estudo com controles adequados.

## 8. Embeddings de fusão: seleção implícita de fontes

A estratégia de *embedding de fusão* concatena vetores de diferentes fontes de informação sobre um POI — localização geográfica (HGI), tempo (Time2Vec) e representação esférica (Sphere2Vec) — em um único vetor de entrada.

Os experimentos revelaram um comportamento inesperado: os vetores HGI têm magnitude 8–15× maior que as demais fontes. Como resultado, o modelo se torna ~90% dependente do HGI já nas primeiras iterações de treino. As demais fontes são praticamente ignoradas.

Mais surpreendente: tentar corrigir esse desequilíbrio via normalização dos vetores individualmente **piorou** os resultados em ~10 pontos de acurácia. A hipótese é que o modelo está fazendo **seleção implícita de fontes por gradiente** — a escala diferente carrega informação semântica real sobre a confiabilidade relativa de cada fonte, e forçar escala igual remove esse sinal.

> **Em aberto:** Se o modelo quase ignora Sphere2Vec e Time2Vec, a fusão é melhor que HGI sozinho? Se não for, as fontes auxiliares adicionam parâmetros sem contribuir sinal. Este é um dos experimentos prioritários.

📄 *Dados:* [`docs/PAPER_FINDINGS.md §Finding 5`](../PAPER_FINDINGS.md)

### 8.1 Diferencial frente às baselines: janela de visita com embeddings por check-in

Uma diferença importante entre a abordagem atual e as baselines da literatura merece ser destacada. A maioria dos trabalhos publicados constrói a janela de histórico de visitas do usuário usando **embeddings em nível de POI** — ou seja, toda vez que o usuário visita o mesmo local, o modelo recebe o mesmo vetor, independente de quando essa visita ocorreu.

Na configuração atual, estamos utilizando **embeddings em nível de check-in** (via Check2HGI): o mesmo POI visitado em momentos diferentes gera vetores diferentes, pois o embedding captura o contexto temporal da visita específica — hora do dia, dia da semana, padrão de mobilidade ao redor daquele evento. Isso representa uma janela de visita semanticamente mais rica: o modelo pode diferenciar "o usuário foi a esse restaurante numa sexta à noite" de "foi num almoço de terça".

**Estado atual:** Em termos teóricos, essa representação deveria ser superior para capturar padrões temporais de mobilidade. Na prática, **ainda não observamos uma melhoria clara** nos experimentos realizados até o momento com essa configuração. Compreender por que a vantagem teórica não se manifesta — e se manifesta após ajuste adequado — é uma das questões abertas do estudo.

## 9. Métricas de avaliação ampliadas: alinhamento com a literatura de recomendação

Uma mudança técnica relevante foi feita no código de avaliação: além de F1 macro e acurácia, o sistema agora calcula e registra automaticamente um conjunto expandido de métricas a cada fold de cada experimento.

**Métricas adicionadas:**

| Métrica | O que mede | Referência na literatura |
|---|---|---|
| **Acc@3 / Acc@5** (Top-K) | Fração de amostras cuja classe correta está no top-K predito | Padrão em next-POI: STRNN, DeepMove, GETNext, Graph-Flashback |
| **MRR** (Mean Reciprocal Rank) | Posição média (inversa) da classe correta no ranking | STAN, GETNext, HMT-GRN, TLR-M |
| **NDCG@3 / NDCG@5** | Qualidade do ranking com desconto logarítmico | LSTPM, CARA, ImNext, MRP-LLM |
| **F1 weighted** | F1 ponderado pelo suporte de cada classe | HAVANA (baseline do nosso grupo) |
| **Accuracy macro** | Acurácia balanceada (média por classe) | Complemento à acurácia micro sob desbalanceamento |

**Por que isso importa para o artigo:** A literatura de next-POI usa predominantemente métricas de ranking (Recall@K, NDCG@K, MRR), enquanto trabalhos de classificação de categoria usam F1 macro e acurácia. Como nosso modelo atua nas duas frentes, reportar apenas F1 deixa a comparação com baselines de recomendação incompleta. Com as métricas já sendo calculadas a cada experimento, o custo de incluí-las na tabela de resultados final é zero — basta selecionar quais reportar na seção de avaliação.

> **Em aberto:** Quais métricas priorizar nas tabelas do artigo? A sugestão atual é: F1 macro (primária, para comparação com categoria) + Acc@5 e NDCG@5 (para comparação com next-POI recommendation) + MRR como secundária.

## 10. Pergunta central ainda em aberto: MTL realmente ajuda?

O artigo do CBIC 2025 concluiu que o MTL não trazia ganho consistente sobre modelos de tarefa única. A hipótese deste trabalho é que essa conclusão era **específica da configuração usada** — a combinação de embeddings DGI + arquitetura FiLM + NashMTL criou um cenário desfavorável ao aprendizado conjunto.

O experimento mais crítico do estudo atual (**Claim C06**) compara diretamente MTL vs. tarefa única na nova configuração (fusão de embeddings + CGC). **Esse experimento ainda não foi executado nos dados corrigidos.**

> **Em aberto:** Se C06 confirmar que MTL ainda não ajuda mesmo na nova configuração, a narrativa do artigo muda de "MTL bem configurado funciona" para "fusão de embeddings específica por tarefa para predição de POI" — ainda publicável, mas exige reposicionamento.

## 10. Baselines reproduzidas: o que já temos para comparar

Além de construir o próprio modelo, reproduzimos três baselines da literatura no mesmo pipeline e dados:

| Baseline | Dataset | F1 macro reportado no paper | F1 macro na nossa reprodução | Gap |
|---|---|---|---|---|
| **HAVANA** (GAT + ARMA) | Florida | 62,9% | 54,2% | −8,7 pp |
| **PGC** | California / Florida / Texas | — | reproduzido | em análise |
| **POI-RGNN** | California / Florida / Texas | — | reproduzido | em análise |

**O gap do HAVANA merece atenção explícita.** A diferença de ~8,7 pontos entre o paper e a nossa reprodução não indica necessariamente que o HAVANA é fraco — reflete uma **assimetria de input**: o HAVANA usa o grafo de POIs completo como entrada, enquanto nossa reprodução usa embeddings pré-treinados. É uma limitação honesta de comparação que precisa ser declarada no artigo, não escondida.

> **Em aberto:** Comparar nossos resultados diretamente contra os números publicados do HAVANA sem mencionar a assimetria de input seria uma imprecisão metodológica. A tabela de resultados do artigo deve comparar contra a **reprodução**, não contra o número do paper.

## 11. Infraestrutura estatística: folds congelados para testes pareados

Um dos experimentos mais importantes para convencer revisores é o **teste de ausência de transferência negativa (C28)**: para cada tarefa individualmente, o MTL precisa ser pelo menos tão bom quanto o modelo de tarefa única, com significância estatística (Wilcoxon signed-rank por fold).

Para que esse teste seja válido, os modelos comparados precisam ter sido treinados e avaliados **exatamente nos mesmos splits**. Qualquer divergência nos folds invalida o teste pareado.

**O que foi implementado:** Um mecanismo de congelamento de folds que registra, para cada combinação de (estado, engine, tarefa), os índices exatos de treino e validação com uma assinatura SHA-256 dos dados de entrada. Toda execução subsequente carrega esse registro e valida que os folds são idênticos antes de treinar.

**Por que importa para o artigo:** Sem folds congelados, uma atualização silenciosa do `sklearn` ou uma regeneração dos parquets pode alterar os splits. O teste pareado (Wilcoxon) perderia validade. Os testes estatísticos de todas as fases P1–P6 dependem desta infraestrutura.

## 12. Limitação metodológica honesta: vazamento nos embeddings upstream

A correção do StratifiedGroupKFold (tópico 2) resolve o vazamento **a jusante** — na divisão dos dados de treinamento do MTLnet. Mas existe um vazamento **a montante** ainda não tratado:

Os embeddings HGI e Check2HGI foram treinados sobre **100% dos check-ins**, sem separação de treino e validação. Isso significa que quando o MTLnet valida num usuário do fold de validação, o embedding desse usuário foi moldado pelos próprios check-ins dele durante o treinamento do HGI.

**Para o artigo:** Esta limitação existe também nos baselines que usamos para comparação (HAVANA, POI-RGNN), portanto a comparação relativa continua válida. É uma limitação honesta a declarar na seção de discussão. A correção completa (treinar embeddings apenas com dados de treino por fold) está documentada como trabalho futuro e está fora de escopo para BRACIS dado o cronograma.

📄 *Referência:* [`docs/issues/DATA_LEAKAGE_ANALYSIS.md`](../issues/DATA_LEAKAGE_ANALYSIS.md)
