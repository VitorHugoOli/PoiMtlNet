## Abstract

This document analyzes a critical data leakage problem identified in the current training pipeline for hierarchical graph embeddings (Check2HGI, HGI) used in multi-task POI prediction. We demonstrate that training embeddings on the full dataset before splitting for downstream task evaluation leads to inflated performance metrics and poor generalization. We propose inductive training strategies with user-based splits for sequential prediction and POI-based splits for category classification, supported by recent literature on graph neural network evaluation methodologies.

## 1. Problem Statement

### Current Issue

The current training pipeline suffers from **data leakage** at the embedding training stage:

1. **Embedding models (Check2HGI/HGI)** are trained on 100% of the check-in/POI data
2. **Downstream models (MTLNet)** then split the data into train/validation folds
3. **Validation samples** use embeddings that were generated by models trained on those same samples

This creates a situation where:
- The embedding model has "seen" validation data during training
- Learned representations encode patterns from the validation set
- Performance metrics are artificially inflated
- True generalization capability is unknown

### Research Question

> How should we structure the data splitting strategy to ensure that embedding models are trained in an **inductive** setting, where validation and test data remain truly unseen during all training phases?

## 2. Background: The Two Prediction Tasks

### 2.1 Next-POI Category Prediction

**Objective**: Given a user's sequence of historical check-ins, predict the category of their next point-of-interest visit.

| Attribute | Value |
|-----------|-------|
| **Input** | Sequence of 9 check-in embeddings `[B, 9, 64]` |
| **Output** | Category prediction (7 classes) |
| **Embedding Type** | Check-in level (Check2HGI) |
| **Features** | Temporal (sin/cos hour/day) + Spatial + Category |
| **Model** | Transformer with causal masking + attention pooling |

**Key Insight**: Each check-in has a unique embedding that captures temporal context. The same POI visited at different times produces different embeddings.

### 2.2 POI Category Classification

**Objective**: Given an embedding of an unknown POI (without category information), classify its category.

| Attribute | Value |
|-----------|-------|
| **Input** | Single POI embedding `[B, 64]` |
| **Output** | Category prediction (7 classes) |
| **Embedding Type** | POI level (HGI/DGI) |
| **Features** | Spatial only (NO category, NO temporal) |
| **Model** | Multi-path MLP with residual connections |

**Key Insight**: This task simulates classifying new POIs that lack category labels (cold-start scenario). The embedding must NOT contain category information.

### 2.3 Comparison

| Aspect | Next-POI Prediction | Category Classification |
|--------|---------------------|------------------------|
| Input granularity | Check-in sequence | Single POI |
| Temporal information | Yes | No |
| Category in input | Yes (for history) | **No** (target variable) |
| Sequential modeling | Yes (Transformer) | No (MLP) |
| Real-world scenario | Predict user behavior | Label new POIs |

## 4. The Data Leakage Problem

### 4.1 Visual Representation

```
┌─────────────────────────────────────────────────────────────────┐
│                    CURRENT PIPELINE (WITH LEAKAGE)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Check-ins: [c1, c2, c3, ..., c80, c81, ..., c100]             │
│              └────── ALL DATA ──────┘                          │
│                         │                                       │
│                         ▼                                       │
│  ┌─────────────────────────────────────────┐                   │
│  │     EMBEDDING TRAINING (Check2HGI)       │                   │
│  │     Trained on c1, c2, ..., c100         │ ← Sees ALL data! │
│  │     Learns patterns from entire dataset  │                   │
│  └─────────────────────────────────────────┘                   │
│                         │                                       │
│                         ▼                                       │
│  Embeddings: [e1, e2, e3, ..., e80, e81, ..., e100]            │
│                         │                                       │
│                         ▼                                       │
│  ┌─────────────────────────────────────────┐                   │
│  │         MTL TRAINING (5-Fold CV)         │                   │
│  │  Train: e1-e80     Val: e81-e100         │                   │
│  │                                          │                   │
│  │  PROBLEM: e81-e100 embeddings were       │                   │
│  │  generated by model trained on c81-c100! │ ← DATA LEAKAGE!  │
│  └─────────────────────────────────────────┘                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 Types of Leakage

Based on the 2025 comprehensive study on data leakage (Joeres et al., 2025):

1. **Preprocessing-related leakage**: The embedding model preprocessing uses the entire dataset to compute statistics (e.g., category distributions, graph structure).

2. **Split-related leakage**: Information leaks between training and evaluation datasets because the split happens AFTER embedding generation.

3. **Transductive leakage**: GNNs in transductive settings can access test node features during training, allowing the model to encode test-specific patterns.

### 4.3 Consequences

| Metric | Current (Leaked) | Reality |
|--------|-----------------|---------|
| Validation F1 | ~0.75 | Unknown (likely 0.60-0.70) |
| Generalization | Appears good | Poor on truly unseen data |
| Model selection | May select overfit model | Unreliable |
| Scientific validity | Compromised | Untrustworthy results |

## 5. Transductive vs Inductive Learning in GNNs

### 5.1 Definitions

**Transductive Learning**:
> "In transductive learning, the model is exposed to both the training and test input data beforehand. The model has access to all nodes in the graph for training and testing. If new nodes are introduced, the model requires retraining." (ResearchGate, 2024)

**Inductive Learning**:
> "In inductive learning, the model only encounters the training data during training. The trained model is then used to generate embeddings for unseen nodes." (ResearchGate, 2024)

### 5.2 Implications for Our Pipeline

| Setting | Current Pipeline | Correct Approach |
|---------|-----------------|------------------|
| Embedding Training | Transductive (all nodes visible) | Inductive (train nodes only) |
| Graph Structure | Full graph | Train-only subgraph |
| Test Data | Visible during training | Hidden until inference |
| Generalization | To seen nodes | To unseen nodes |

### 5.3 Evidence from Literature

From Hamilton et al. (2017) - GraphSAGE:
> "We present GraphSAGE, a general inductive framework that leverages node feature information to efficiently generate embeddings for previously unseen nodes."

Key insight: **Inductive methods learn aggregation functions, not node-specific embeddings**, enabling generalization to new nodes.

## 6. Literature Review: Best Practices

### 6.1 Data Splitting for Sequential Recommendation

From "Time to Split" (RecSys'25):

| Strategy | Description | Recommendation |
|----------|-------------|----------------|
| Leave-One-Out | Last item per user for test | Causes temporal leakage |
| User Temporal | Last X% per user | Better, but still leaks |
| **Global Temporal** | All data after time T | **Most realistic** |

> "Global temporal splitting addresses temporal leakage by evaluating on distinct future periods. Researchers should evaluate using temporal global splitting, the most accepted realistic setting."

### 6.2 Embedding Training Best Practices

From "Information Leakage in Embedding Models" (CCS 2020):

> "Pre-training embeddings on a large amount of unlabeled data and fine-tuning them for downstream tasks is now a de facto standard. However, adversaries with access to embeddings can perform information leakage attacks."

Implication: If embeddings are trained on test data, they may encode test-specific information that leaks to downstream models.

### 6.3 Cold-Start POI Classification

From IJCAI 2024 on POI Category Embeddings:

> "The POI Category Embedding problem involves learning hierarchy-enhanced representations where categories combine with their upper-level ancestors via attention mechanisms."

Key insight: For category classification, embeddings should capture **hierarchical spatial patterns**, not category-specific features (which would defeat the purpose).

## 7. Proposed Solutions

### 7.1 Solution A: User-Based Inductive Split (For Next-POI Task)

**Rationale**: Keep user trajectories intact; test generalization to new users.

```
┌────────────────────────────────────────────────────────────────┐
│                CORRECTED PIPELINE (USER-BASED)                 │
├────────────────────────────────────────────────────────────────┤
│  Step 1: Split USERS first                                     │
│  ├─ Train users: 80%                                           │
│  ├─ Val users: 10%                                             │
│  └─ Test users: 10%                                            │
│                                                                │
│  Step 2: Train embeddings on TRAIN USERS ONLY                  │
│  ├─ Build graph from train user check-ins                      │
│  ├─ Train Check2HGI on this subgraph                           │
│  └─ Save model checkpoint                                      │
│                                                                │
│  Step 3: INFERENCE for val/test users                          │
│  ├─ For each val/test user:                                    │
│  │   ├─ Add their check-ins to graph                           │
│  │   ├─ Forward pass (no gradient!)                            │
│  │   └─ Extract embeddings                                     │
│  └─ Save val/test embeddings separately                        │
│                                                                │
│  Step 4: Train MTL on train embeddings                         │
│  ├─ Validate on val embeddings                                 │
│  └─ Test on test embeddings (final evaluation)                 │
└────────────────────────────────────────────────────────────────┘
```

**Pseudocode**:

```python
from sklearn.model_selection import train_test_split

# Step 1: Split users
unique_users = checkins['userid'].unique()
train_users, temp_users = train_test_split(unique_users, test_size=0.2, random_state=42)
val_users, test_users = train_test_split(temp_users, test_size=0.5, random_state=42)

# Step 2: Train embeddings on train users only
train_checkins = checkins[checkins['userid'].isin(train_users)]
model = train_check2hgi(city, args, checkins=train_checkins)

# Step 3: Inference for val/test users
val_checkins = checkins[checkins['userid'].isin(val_users)]
val_embeddings = inference_check2hgi(model, checkins=val_checkins)

test_checkins = checkins[checkins['userid'].isin(test_users)]
test_embeddings = inference_check2hgi(model, checkins=test_checkins)

# Step 4: Train MTL with proper splits
train_sequences = create_sequences(train_embeddings)
val_sequences = create_sequences(val_embeddings)
test_sequences = create_sequences(test_embeddings)

mtl_model = train_mtl(train_sequences, val_sequences)
evaluate_mtl(mtl_model, test_sequences)  # Final evaluation
```

### 7.2 Solution B: POI-Based Split (For Category Classification Task)

**Rationale**: Test if model can classify truly unseen POIs (cold-start scenario).

```
┌─────────────────────────────────────────────────────────────────┐
│                    CORRECTED PIPELINE (POI-BASED)               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Step 1: Split POIs (stratified by category)                   │
│  ├─ Train POIs: 80%                                            │
│  └─ Test POIs: 20%                                             │
│                                                                 │
│  Step 2: Train HGI embeddings on TRAIN POIs ONLY               │
│  ├─ Build spatial graph from train POIs                        │
│  ├─ Train HGI on this subgraph                                 │
│  └─ Save model checkpoint                                      │
│                                                                 │
│  Step 3: INFERENCE for test POIs                               │
│  ├─ Add test POIs to graph                                     │
│  ├─ Forward pass (no gradient!)                                │
│  └─ Extract embeddings                                         │
│                                                                 │
│  Step 4: Train category classifier on train POI embeddings     │
│  └─ Test: Can it classify UNSEEN POIs?                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Why POI-based for category task?**:
- The goal is classifying unknown/new POIs
- Must test on POIs never seen during embedding training
- Simulates real-world cold-start scenario
- Different from Next-POI which is about user behavior

### 7.3 Comparison of Approaches

| Approach | Best For | Split By | Advantages | Disadvantages |
|----------|----------|----------|------------|---------------|
| User-based | Next-POI | Users | Keeps trajectories intact | May have POI overlap |
| POI-based | Category | POIs | True cold-start test | Loses user context |
| Temporal | Production sim. | Time | Most realistic | Less training data |

### 8.2 Key Implementation Details

#### 8.2.1 Inference Mode for GNNs

The embedding model must support **inference mode** for unseen nodes:

```python
def inference_check2hgi(model, new_checkins, existing_graph):
    """
    Generate embeddings for new check-ins without training.

    Args:
        model: Trained Check2HGI model
        new_checkins: DataFrame of new check-ins (val/test users)
        existing_graph: Graph from training phase

    Returns:
        embeddings: Embeddings for new check-ins
    """
    model.eval()
    with torch.no_grad():
        # Add new nodes to graph
        extended_graph = add_nodes_to_graph(existing_graph, new_checkins)

        # Forward pass only
        embeddings = model.encode(extended_graph)

        # Extract only new node embeddings
        new_embeddings = embeddings[len(existing_graph.x):]

    return new_embeddings
```

#### 8.2.2 Handling Graph Extension

For inductive inference, we need to:
1. Keep the trained model weights frozen
2. Add new nodes to the graph structure
3. Compute embeddings via message passing from trained neighbors
4. Extract only the new node embeddings

## Appendix B: Glossary

- **Check-in**: A record of a user visiting a POI at a specific time
- **POI**: Point of Interest (e.g., restaurant, park, shop)
- **Transductive**: Learning setting where test nodes are visible during training
- **Inductive**: Learning setting where test nodes are unseen during training
- **Data Leakage**: Unintended information flow from test to training data
- **Cold-Start**: Scenario where model must handle unseen items/users
- **GNN**: Graph Neural Network
- **MTL**: Multi-Task Learning
