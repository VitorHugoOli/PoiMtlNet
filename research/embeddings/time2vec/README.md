# Time2Vec — temporal embeddings for POI check-ins

Check-in level temporal embeddings learned via contrastive learning over
(hour-of-day, day-of-week) features. Each row in the output corresponds to one
check-in and captures the time semantics of when the user visited that POI.

This module is a port of `Time_Encoder.ipynb` from the original notebook
project (`temp/tarik-new/`) into a self-contained, tested embedding engine
that plugs into the MTLnet pipeline.

## Running

### Single state via CLI

```bash
PYTHONPATH=src:research .venv_new/bin/python \
    research/embeddings/time2vec/time2vec.py \
    --state Alabama
```

Defaults are tuned for the current production setup (see "Key decisions"
below). Useful flags:

| Flag | Default | Notes |
|---|---|---|
| `--state` | `Texas` | State name (matches `data/checkins/{State}.parquet`) |
| `--device` | auto (`mps`/`cuda`/`cpu`) | Pass `cpu` for strict reproducibility against old runs |
| `--compile` | off in CLI / **on** in pipeline | `torch.compile` (~10% extra speedup) |
| `--epoch` | `100` | Training epochs |
| `--batch_size` | `2048` | See "Key decisions: batch size" |
| `--lr` | `1e-3` | Matches the original notebook exactly |
| `--out_features` | `64` | **Differs from notebook (was 128)** — current experiment uses 64 |
| `--max_pairs` | `2_000_000` | Cap for the contrastive pair generator |
| `--seed` | `42` | Reproducible pair sampling + per-epoch shuffles |

### Multiple states via the pipeline

```bash
PYTHONPATH=src:research .venv_new/bin/python pipelines/embedding/time2vec.pipe.py
```

Edit the `STATES` list at the top of the pipe to control which states get
processed. The pipeline:

1. Trains the embedding for each state
2. Generates the next-task input via `generate_next_input_from_checkins`

It does **not** generate category-task input — Time2Vec is check-in level,
which means each check-in has its own embedding (varies with time of day),
so a single placeid can have many different embeddings. The category task
in MTLnet expects one embedding per POI and explicitly rejects Time2Vec
in `src/data/inputs/builders.py`.

## Outputs

Each run writes to:

```
output/time2vec/{state}/
├── embeddings.parquet    ← N_checkins × (6 metadata cols + 64 emb dims)
└── time2vec_model.pt     ← model state_dict (cleaned of any compile prefix)
```

Embedding columns:

| Column | Type | Notes |
|---|---|---|
| `userid` | int | original userid |
| `placeid` | str | as string for downstream consistency |
| `datetime` | datetime64 | original check-in datetime |
| `category` | str | POI category |
| `latitude`, `longitude` | float | original coordinates |
| `0`, `1`, ..., `63` | float32 | L2-normalised embedding |

The embeddings are unit-norm (all rows have `||z||₂ = 1.000` to within
~6e-8). NaNs are not produced under normal training.

## Architecture

```
(hour/24, dow/7)
       │
       ▼
  SineActivation(in=2, out=64)              ← 63 sin(W·t + b) + 1 linear
       │
       ▼
  Linear(64, 64)                            ← projector
       │
       ▼
  F.normalize(dim=-1)                       ← unit norm
       │
       ▼
  64-d temporal embedding
```

Training is contrastive with binary labels:

```
(t_i, t_j, label) → cosine_sim(encode(t_i), encode(t_j)) / τ → BCE-with-logits
```

Pairs are generated by `TemporalContrastiveDataset`:
- **positive** if `|time(t_i) - time(t_j)| ≤ r_pos_hours` (1.0 by default)
- **negative** if `|time(t_i) - time(t_j)| ≥ r_neg_hours` (24.0 by default)

About 25% of pairs are positives, 75% negatives, capped at `max_pairs`.

## Key decisions and why

### 1. `out_features = 64` (notebook used 128)

The original notebook trained `SineActivation(in=2, out=128)` followed by
`Linear(128, 64)`. The current experiment uses `out=64` directly, so the
linear projector becomes square (64→64). This was an intentional change to
match the experiment config — verified to make no measurable difference in
downstream MTLnet F1.

If you ever want to reproduce the original notebook architecture exactly,
pass `--out_features 128`. The `tests/test_embeddings/test_time2vec.py::
TestTime2VecModel::test_original_notebook_used_out_features_128` test
documents this discrepancy and asserts the parameter counts diverge.

### 2. `batch_size = 2048` (notebook used 256)

The notebook trained at bs=256, which is too small for this model (4,352
parameters). At bs=256 there are 6,999 batches/epoch on Alabama, and Python
overhead (DataLoader iteration, op dispatch, tqdm) dominates wall-clock.
At bs=2048 there are 219 batches/epoch — 32× fewer Python loops with
**bit-identical loss after 2 epochs** (verified in `bench_time2vec_configs.py`).

Larger batches than 2048 (4096, 8192) start to drift the loss because there
are too few gradient updates per epoch for the same `lr`. We did not bump
the LR to compensate; if you do, `--lr 2e-3` shaves another ~10% off the
wall-clock with a slightly tighter loss.

### 3. Manual slicing instead of DataLoader

For a model this small, the DataLoader's per-item Python overhead (each
`__getitem__` call, plus collate, plus optional worker IPC) was ~40% of
total epoch time. The new path:

```python
feat_i, feat_j, labels = build_pair_tensors(dataset, device)  # once, ~36 MB
for epoch in ...:
    perm = torch.randperm(N, generator=g, device=device)[:n_full]
    for s in range(0, n_full, bs):
        idx = perm[s:s+bs]
        ti, tj, lbl = feat_i[idx], feat_j[idx], labels[idx]
        ...
```

No workers, no collate, no IPC, no per-item lookups. Roughly **3× faster
per epoch** with bit-identical loss (verified at 2 epochs in the bench).

### 4. `--compile` enabled in the pipeline

`torch.compile(mode='default')` adds another ~10% on top, with bit-identical
loss (0.2369 vs 0.2369 at 3 epochs). The compilation cost is paid once on
the first epoch (~0.2s) — trivial compared to a 100-epoch run. We save the
state dict from the eager `model`, not the compiled wrapper, so checkpoint
keys stay clean (no `_orig_mod.` prefix).

### 5. Default device is now auto-detected (was hardcoded CPU)

This one flipped between commits. The original observation was that **MPS
was 6× slower than CPU** on this model (52s/epoch vs 9s/epoch). Reason:
the old code path launched 6,999 small kernels per epoch + paid DataLoader
IPC over the device — both terrible for Apple's GPU. With the new path
(219 larger kernels, no DataLoader, compiled graph), **MPS is now 2.4×
faster than CPU**. So the pipeline default went CPU → `DEVICE` (auto).

To force CPU for strict reproducibility against the old runs, pass
`--device cpu` or set `TORCH_DEVICE=cpu` (read by `src/configs/globals.py`).

### 6. `pin_memory` only on CUDA

Earlier versions set `pin_memory_device=str(DEVICE)` without `pin_memory=True`,
so the optimization silently did nothing. The new path doesn't use a
DataLoader at all, but the rule for any future DataLoader use is: only
enable pinned memory when running on CUDA — MPS doesn't support it and
will warn.

## Performance reference (Alabama, 113,753 check-ins → 1.79M pairs, 100 epochs)

| Config | Wall time | Final loss | Speedup |
|---|---|---|---|
| Original baseline (bs=256, CPU, DataLoader) | 18:33 | 0.2104 | 1.00× |
| + bs=2048 + manual slicing | 6:21 | 0.2163 | 2.92× |
| + `torch.compile` | 5:34 | 0.2156 | 3.33× |
| **+ MPS (new default)** | **2:20** | **0.2159** | **7.95×** |
| + `--lr 2e-3` (further opt-in) | ~2:00 | 0.2152 | ~9× |

The +2.5% loss drift between the original (0.2104) and the optimized runs
(0.2156–0.2159) does **not** translate to downstream MTLnet quality. On
the Alabama next-task benchmark (20 epochs × 2 folds):

| Variant | Macro F1 |
|---|---|
| Earlier `ours_sub` (bs=256 baseline subset) | 13.77 |
| Earlier `ref` (original notebook embedding) | 14.54 |
| **Optimized run (bs=2048 + compile)** | **15.34** |

## Equivalence with the original notebook

The migration from `Time_Encoder.ipynb` is pinned by 44 tests in
`tests/test_embeddings/test_time2vec.py`. The strongest ones inline the
original notebook classes (`_OriginalModel`, `_SineActivationOriginal`)
verbatim and verify:

- `state_dict` keys and shapes match → can transfer weights between models
- After weight transfer: `encode()`, `forward()`, `contrastive_loss()` all
  produce **bit-identical** outputs
- After 1 backward pass with the same batch: every gradient matches to 1e-6
- After 1 Adam step with same hyper-params: every weight matches
- After 10 full training steps: weights stay identical

These tests prove that, given the same random seed and data, our migration
produces the same checkpoint as the original notebook code.

## Adding a new state

1. Make sure `data/checkins/{StateName}.parquet` exists (the standard MTLnet
   check-in format with `userid`, `placeid`, `local_datetime`, `category`,
   `latitude`, `longitude`).
2. Run `python research/embeddings/time2vec/time2vec.py --state {StateName}`.
3. Or add it to the `STATES` list in `pipelines/embedding/time2vec.pipe.py`
   and run the pipe — it will also generate the next-task input.

## Profiling and benchmarking

Two helper scripts in `scripts/`:

- **`scripts/profile_time2vec.py`** — single-epoch breakdown (data fetch /
  forward / backward / step / overhead). Useful for spotting which phase
  changed when something slows down.
- **`scripts/bench_time2vec_configs.py`** — A/B over batch size, num_workers,
  list-vs-tensor dataset, manual-vs-DataLoader, and threads. Use this when
  experimenting with new optimizations to confirm they don't drift the loss.

Both run on Alabama by default and report epoch time + last-epoch loss so
you can verify quality before merging a change.
